1/27/20: Learn to setup Putty for easier remote Linux server administration (Windows users only)

If you use Windows and are administering a remote Linux server on Amazon Web Services (AWS) using Windows' shitty command-line shell (yes, that includes PowerShell), it's time you learned to switch to the most popular SSH/Telnet client called PuTTY (download the 64-bit version here). This will save you from having to type in that long-ass command to SSH into your remote server every time (you know the one I'm talking about: ssh -i <the-location-of-my-pem-file> <my-user-name>@<ip-address-of-server>). After you have installed PuTTY, open the program and type in the IP address of your server in the "Host Name (or IP address)" field, followed by a convenient name to remember this login by in the "Saved Sessions" field. Now click the "Save" button (I've highlighted all those areas yellow in "Figure 1").

The next step is to convert the .pem key file that you got from AWS into a format that PuTTY can understand. If you right click on the open PuTTY program from the Windows task bar at the bottom, it will pull up a sub-menu list, which includes an entry entitled "Run PuTTYgen". Click on that to open up the PuTTY key generator. Once open, click on the "Conversions" menu at the top, then "Import key". Locate your .pem file to load it into the program, then click on the "Save private key" button (all these areas have been highlighted yellow for you in "Figure 2"). If it warns you about not setting a password, that is your choice to ignore or not. If you do set a password on the key, then you'll always be prompted to type it in every time you log into your remote server; otherwise you'll just be logged in automatically instead. Whichever option you choose, you now have to save your new private key for PuTTY in a safe location on your computer (this key file will end with the extension .ppk).

The final step is to configure PuTTY to use this new key file. Hopefully you still have your PuTTY client open from the first step. Refer back to the image on the right: you'll notice I have also highlighted yellow 3 sub-menus on the left-hand side of the image (namely, "Connection", "SSH", and "Auth"). Expand all 3 of those menus, and select the "Auth" sub-menu. You'll now notice a "Browse" button, which you will click on in order to select the .ppk key file you created in the previous step. Once you have selected that file, go back to the "Session" sub-menu of PuTTY, make sure the name you picked for this session is in the box under "Saved Sessions", and click on the "Save" button again so that it will associate that key file with this particular session. Now simply double-click on the name of your saved session in the box to launch the session, click "Yes" on the popup warning, and you should be in!

1/26/20: Use bash aliases to speed things up in the Linux terminal!

One of very first things I do when I set up a Linux server is to set up my Bash aliases which increases productivity since I end up being able to do more by typing less and with much more speed. As an example, instead of typing sudo apt update everytime you want to update your software repositories, imagine being able to run the same command by just typing update. That's where bash aliases come into play. Here's how to set them up: on your Linux server, first type cd to ensure that you are in your home directory. Then type ls -lah to reveal all your hidden files (these are files whose names begin with a period, such as ".bashrc"). In fact, we are going to be opening the ".bashrc" file to make sure the option for bash aliases is enabled, so fire up the nano text editor by typing nano .bashrc. Scroll down to the bottom of the file and make sure the second to last code block with the declaration shown in the image below 

              
if [ -f ~/.bash_aliases ]; then
. ~/.bash_aliases
fi
is uncommented (meaning there are no "#" to the left of it). By default, Ubuntu Server has this block of code uncommented, so if this is the case for you, just exit nano by pressing Ctrl+X; otherwise delete the strip of '#'s to the left of it, type Ctrl+O to save, followed by Ctrl+X to exit nano.

Now, let's create the bash aliases file by typing nano .bash_aliases. Inside, this file will contain your alias definitions. The syntax for this is: alias <your-alias>="<actual-command>". In the provided image above, I have included some examples of my alias definitions. The first two are aliases for the same command, namely, cd ... With these aliases, I can now type .. or cd.. (without the space between "cd" and "..") to go up one directory, or if I need to reboot the server, I can now just type reboot instead of sudo shutdown -r now. Once you have your definitions set, type Ctrl+O to save your changes and Ctrl+X to exit nano. In order to have your alias definitions take effect without rebooting your machine or logging out and back in, type source .bash_aliases. This command will tell the bash shell to parse your alias definitions so that you can start using them right away!

1/25/20: How to use nginx to rapidly deploy other pages of your website

For first time web developers, once you have a web server up and running for the first time ever, it can be very exciting. It's important, however, to make sure you have it configured properly before you jump right into development. While there are many ways in setting up nginx (the same goes for Apache2), one method I have always found very convenient and easy to execute is enabling multiple locations in the nginx configuration file for the different areas of my website. What this means is, if I add, for example, location "/photos" to my config file, then users will then be able to navigate to "www.mywebsite.com/photos/". I have seen other people preach creating C-NAME DNS records in order to create sub-domains that point to other areas of your website (in this case, using the previous example, it would point to "photos.mywebsite.com"), but I feel that this is unnecessary work, not to mention that sub-domains were not meant to be used in that way.

In order to add these locations to your nginx configuration file, do the following: in your web server on the command line, type in cd /etc/nginx/sites-available/ then press enter. Then locate whichever website configuration file you wish to edit, and type sudo nano <your-config-file-name-here> then press enter. Locate the block of code in which the word location followed by a forward-slash appears for the first time (highlighted yellow in the image), and add underneath that declaration another block of code starting with the word location as well (refer to the example in the image). As you can see, I have added two location entries, with each location value preceeded by a forward-slash (in this case, "/memorymatch" and "/main"). Make sure you do not forget to add the forward-slash in front of your location value as well! Now, simply copy what I have done in the image, by typing the left curly brace, inside of which you will type the property "autoindex" followed by a value of "on" and a semicolon, then close it with a right curly brace. Of course, proper indentation is recommended. In case you were wondering, autoindex tells nginx to automatically locate any files named index.* and to serve up that file when users navigate to that location of your website.

Once you have finished adding all your locations, type CTRL+O to save your changes, followed by CTRL+X to exit nano. You will need to restart nginx by typing sudo service nginx restart and pressing enter. Now users will be able to navigate to other pages of your website by appending those location names to the end of your website name (just make sure you have index.html files in those folders on your webserver!).
